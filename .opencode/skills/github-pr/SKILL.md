# GitHub PR 제출 절차 (Spring Boot MCP 프로젝트용)

이 skill은 코드 변경이 완료되고 모든 검증 게이트를 통과한 후 PR을 생성할 때 적용한다.

본 프로젝트는 다음 특성을 가진다:

- Spring Boot 3.x
- Java 17+
- Spring AI MCP Server
- Streamable HTTP 기반 MCP
- Claude Desktop 연동
- 기존 사내 시스템 wrapper
- Docker 기반 실행 환경

따라서 일반 Node.js 웹앱 기준 절차가 아닌 MCP 서버 기준 검증을 수행해야 한다.

# 사전 체크 (모든 항목 PASS 필요)

## Build / Test Gate

- [ ] Gate 1 (컴파일) 통과
```bash
mvn clean compile
````

* [ ] Gate 2 (테스트) 통과

```bash

```

* [ ] Gate 3 (패키징) 통과


* [ ] (사용 중인 경우) 코드 스타일 검사 통과

```bash

```

또는

```bash
mvn checkstyle:check
```

# 브랜치 전략

다음 패턴 유지:

```text
feature/<task-id>-<short-desc>
fix/<task-id>-<short-desc>
refactor/<task-id>-<short-desc>
chore/<task-id>-<short-desc>
docs/<task-id>-<short-desc>
```

예시:

```text
feature/MCP-12-check-availability
fix/MCP-15-user-context-leak
refactor/MCP-20-backend-api-client
chore/MCP-30-docker-config
```

# Commit 규칙

## 컨벤션

```text
feat: 기능 추가
fix: 버그 수정
refactor: 리팩토링
chore: 설정 변경
docs: 문서 수정
test: 테스트 추가/수정
```

scope 사용 가능:

```text
feat(mcp): check_availability 도구 추가
fix(client): referer 헤더 누락 수정
refactor(service): 예약 검증 로직 분리
```

---

## 규칙

* 반드시 한국어로 작성
* 하나의 commit에는 하나의 논리적 변경만 포함
* 의미 없는 메시지 금지
* WIP commit 금지
* auto-generated message 금지

---

## 예시

```text
✅ feat: 회의실 가용성 조회 도구 추가
✅ fix: UserContext ThreadLocal 누수 문제 수정
✅ refactor: BackendApiClient 응답 처리 공통화
✅ test: 예약 취소 API 통합 테스트 추가

❌ fix: 수정
❌ update
❌ wip
```

---

## 본문이 필요한 경우

```text
fix: 미래 날짜 예약 조회 시 dt 충돌 문제 수정

기존 태블릿 endpoint는 내부 forward 시 현재 날짜를 강제로
주입하고 있어 미래 날짜 조회가 정상 동작하지 않았다.

이제 우선 태블릿 endpoint를 시도하고,
응답 이상 시 실제 controller endpoint로 fallback 하도록 수정했다.

Resolves: MCP-22
```

# MCP 프로젝트 특화 주의사항

## ThreadLocal 누수

`UserContext.clear()` 누락 시 사용자 간 사번 오염 발생 가능.

반드시 interceptor `afterCompletion()`에서 clear 수행.

---

## Referer 검증

기존 시스템 referer 검증 의존 구조를 변경하지 말 것.

새 endpoint 직접 호출 시:

* referer 검증 여부
* 인증 여부
* fallback 전략

반드시 확인.

---

## Forward URL dt 충돌

다음 endpoint는 내부 forward 시 dt 충돌 가능성 존재:

* `selectMtgRmList.do`
* `selectResveList.do`

미래 날짜 테스트 필수.

---

## Destructive Tool

다음 tool은 반드시 사용자 확인 후 호출:

* `create_reservation`
* `cancel_reservation`

tool description에 확인 절차가 반드시 포함되어야 함.

* DTO 필드명 mismatch
* multipart/form-data 누락
* UTF-8 한글 깨짐
* ThreadLocal clear 누락
* MCP Inspector 테스트 생략
* Docker 환경변수 누락
* 예약 충돌 오류 미처리
